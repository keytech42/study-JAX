# JAX Deep Dive: `jnp.dot` vs `jnp.matmul` (선형대수 vs 텐서 기하학)

이 문서는 JAX(및 NumPy)에서 행렬 연산을 수행할 때 가장 흔히 혼동되는 두 함수, `jnp.dot`과 `jnp.matmul`(`@` 연산자)의 미묘한 차이를 해부한다.

단순히 "행렬 곱셈"이라는 결과에 집중하는 것이 아니라, **차원(Dimension)이 늘어날 때 두 함수가 데이터를 해석하는 철학적 차이**를 분석하고, 왜 현대 딥러닝 아키텍처에서는 `jnp.matmul`이 표준으로 자리 잡았는지 수학적으로 증명한다.

---

## 1. 핵심 문제: "곱한다"는 행위의 중의성

입력이 2차원 행렬일 때 두 함수의 결과는 동일하다. 하지만 입력이 3차원 이상의 텐서(Tensor)가 되는 순간, 두 함수는 완전히 다른 길을 걷는다.

**문제의 본질:** 데이터가 "배치(Batch)"로 묶여 있을 때, 우리는 마지막 두 차원(행렬)만 곱하고 싶은 것인가, 아니면 모든 차원을 가로질러 복잡한 내적(Inner Product)을 수행하고 싶은 것인가?

---

## 2. 멘탈 모델과 비교 철학: 수축(Contraction) vs 배치(Batching)

이 두 함수의 차이를 이해하려면 "데이터를 바라보는 관점"을 정의해야 한다.

!!! abstract "jnp.dot: 선형대수의 수축 (Contraction)"
    *   **철학:** 두 텐서의 특정 축(Axis)을 "맞물려 소거(Contract)"하는 정통 선형대수 방식이다.
    *   **동작:** `dot(A, B)`는 A의 **마지막 축**과 B의 **뒤에서 두 번째 축**을 곱해서 더한다.
    *   **특징:** 배치의 개념이 없다. 앞선 차원들이 많아지면 그 모든 조합에 대해 연산을 수행하려 하므로 결과 텐서의 크기가 예상보다 거대해질 수 있다.

!!! abstract "jnp.matmul (@): 텐서 기하학의 배치 (Batching)"
    *   **철학:** 마지막 두 차원을 제외한 나머지 차원을 **"단순한 묶음(Batch)"**으로 취급한다.
    *   **동작:** 마지막 두 축에 대해서만 행렬 곱셈을 수행하고, 나머지 앞쪽 차원들은 브로드캐스팅(Broadcasting) 규칙에 따라 평행하게 처리한다.
    *   **특징:** 현대 딥러닝의 "Batch Matrix Multiplication" 직관에 완벽히 부합한다.

---

## 3. 심층 기전: 차원별 동작 비교 분석

### 케이스 1: 2D 행렬 (Matrix-Matrix)
*   **둘 다 동일:** $C = A \times B$ 연산을 수행한다.
*   $A: (M, K), B: (K, N) \rightarrow C: (M, N)$

### 케이스 2: 3D 이상의 고차원 (The Batching Trap)
가장 큰 차이가 발생하는 지점이다. 배치 크기가 $B$인 두 텐서를 곱한다고 가정하자.

!!! question "PyTorch는 어떻게 할까?"
    PyTorch는 이 혼란을 피하기 위해 매우 엄격한 라이브러리 설계를 택했다.
    *   `torch.dot`: 오직 1D 벡터의 내적만 허용한다. 행렬을 넣으면 에러가 난다.
    *   `torch.mm`: 오직 2D 행렬 곱셈만 허용한다. 배치가 있으면 에러가 난다.
    *   `torch.matmul` / `@`: JAX의 `jnp.matmul`과 동일하게 배치 브로드캐스팅을 지원한다.
    *   `torch.bmm`: 명시적으로 3D(Batch) 입력을 요구하여 모호성을 원천 차단한다.

JAX/NumPy는 역사적 이유로 `dot`에 매우 관대한 유연성을 부여했는데, 이것이 오히려 독이 되는 경우가 많다.

```python
import jax.numpy as jnp

# Batch size=10, Matrix size=(3, 4) & (4, 5)
A = jnp.ones((10, 3, 4))
B = jnp.ones((10, 4, 5))

# 1. jnp.matmul (추천)
res_matmul = jnp.matmul(A, B) # (1)
# 결과: (10, 3, 5) -> 각 배치별로 독립적인 행렬곱 수행

# 2. jnp.dot (주의)
res_dot = jnp.dot(A, B) # (2)
# 결과: (10, 3, 10, 5) -> 재앙 발생!
```

1. **배치 처리:** 앞의 `10`이라는 숫자를 "병렬적인 묶음"으로 인식한다. $10$개의 $(3 \times 5)$ 행렬이 생성된다.
2. **축 수축:** `A`의 마지막 축($4$)과 `B`의 뒤에서 두 번째 축($4$)은 맞물리지만, `A`의 첫 번째 축($10$)과 `B`의 첫 번째 축($10$)을 독립적인 차원으로 유지하며 결과에 모두 포함시켜 버린다. 이를 **외적(Outer Product)의 성질**이 섞였다고 표현한다.

## 4. 구체적 해결책: 왜 `@`를 써야 하는가?

딥러닝 아키텍처(특히 트랜스포머의 어텐션)에서 우리는 항상 `(Batch, Heads, Seq, Hidden)` 형태의 데이터를 다룬다.

1.  **가독성과 의도 명확화:** `a @ b`를 사용하면 코드를 읽는 사람에게 "나는 지금 마지막 두 차원을 행렬로 보고 곱하고 있다"는 의도를 즉각 전달한다.
2.  **브로드캐스팅 안전성:** `matmul`은 차원이 맞지 않으면 에러를 내거나 명확한 브로드캐스팅 규칙(예: `(10, 3, 4) @ (4, 5) -> (10, 3, 5)`)을 적용하지만, `dot`은 의도치 않은 거대 텐서를 만들어 메모리 부족(OOM)을 일으킬 위험이 크다.
3.  **XLA 최적화:** JAX 내부적으로 두 함수는 결국 `DotGeneral`이라는 저수준 XLA 연산으로 번역된다. 하지만 `matmul` 구조가 XLA 컴파일러가 행렬 연산 유닛(TPU의 MXU 등)을 활용하도록 최적화된 형태로 레이아웃을 잡기에 더 유리하다.

## 5. 결론: 실전 가이드

| 상황 | 추천 연산 | 이유 |
| :--- | :--- | :--- |
| **단순 벡터 내적** | `jnp.dot(v1, v2)` | 전통적인 내적 기호로 직관적임 |
| **2D 행렬 곱셈** | `A @ B` 또는 `jnp.matmul` | 깔끔하고 현대적임 |
| **딥러닝 배치 연산** | **무조건 `A @ B`** | 배치 차원을 안전하게 보존함 |
| **특정 축 지정 연산** | `jnp.einsum` | `dot`보다 훨씬 명시적으로 축을 제어함 |

!!! info "제약(Constraint) vs 관례(Convention): einsum의 대두"
    사실 현대 고수준 JAX 코드에서는 `dot`이나 `matmul`조차 생략하고 `jnp.einsum('...ij,...jk->...ik', A, B)` 형식을 쓰는 경우가 많다. 이는 "어떤 축이 소거되는지"를 문자로 명시하기 때문에, `dot`의 모호함과 `matmul`의 배치 규칙을 모두 아우르는 가장 안전한 **시스템적 제약**으로 작동하기 때문이다.

---

*[Broadcasting]: 모양이 다른 텐서들 간의 연산을 가능하게 하는 NumPy/JAX의 규칙. 작은 텐서를 큰 텐서의 모양에 맞춰 "가상으로" 확장한다.
*[Contraction]: 텐서 연산에서 특정 지수(Index)를 공유하며 합산하여 차원을 줄이는 행위.
*[MXU]: Matrix eXecution Unit. Google TPU에서 행렬 곱셈을 초고속으로 처리하는 전용 하드웨어 가속기.
